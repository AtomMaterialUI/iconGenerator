## Heuristic used to classify "icon request" issues
An issue is considered an icon request when **any** of these signals is present:
1. Title contains `Icon Request` (for example, `[Icon Request]: Redis`).
2. Labels include an icon-request marker (for example, names containing `icon request` / `icon`).
3. Body matches the icon-request template sections (`### Icon Type` + `### Folder names` / `### File names` / `### File extensions` / `### Language IDs`).
4. Legacy issue format explicitly listing `# Icons needed`.

## How association presence was checked
- Searched all `*_associations.xml` files in this repo: `icon_associations.xml` and `folder_associations.xml`.
- Folder-name requests were checked against `folder_associations.xml`; file/extension/language requests were checked against `icon_associations.xml`.
- Matching used token-based case-insensitive string/regex-term presence checks against association attributes/patterns.
- Partial matches or non-structured requests are marked as `needs manual verification`.
